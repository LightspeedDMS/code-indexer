**Status:** FINAL — all design decisions locked (revision pass 4: precision AC fixes to the apply path — C1 explicit per-poll restart-check, C2 applied-worker-count resolver, M1 DEPLOY-never-TARGET + corrupt handling, M2 shared file co-location, M3 bump-no-version-advance, M4 solo-CLI `--workers`, M5 applied-write sequencing, targeted generation re-inject; architecture affirmed sound by both pass-3 reviewers, NOT a redesign). Ready for `/write-epic-spec`.
**Supersedes:** v10.160.0 interim fix (restart-signal handler re-applies `_ensure_workers_config` from local `config.json`).

## 1. Problem

Launch-affecting settings (`workers`, `log_level`, `host`, `port`) live in the **per-node bootstrap `config.json`**. In a cluster (HAProxy + N nodes + shared PostgreSQL), editing them in the Web UI changes only the one node the load balancer routed to — they do **not** propagate. This violates the project's **"Cluster-Aware State — ABSOLUTE RULE."**

Investigation also found a related latent bug: in the **systemd/auto-updater deployment path** (`installer.py:208`: `ExecStart={python_exe} -m code_indexer.server.main --host 0.0.0.0 --port {port}`), `--host` is hardcoded and `--port` is baked at install time, and the auto-updater's *only* ExecStart re-writer is `_ensure_workers_config` (`--workers`). There is **no `_ensure_host`/`_ensure_port`**. So editing `host`/`port` in the config screen is a **no-op in every systemd deployment (staging, production, all clusters)** — functional only in the solo CLI-managed path (`server_lifecycle_manager._start_server_process:377`, which reads `config["host"]`/`config["port"]`). The UI shows them editable + "restart required," which is misleading. This epic fixes that too.

**Two facts surfaced in revision pass 3 reshape the design:**

1. **`log_level` does NOT belong in ExecStart / uvicorn (CRITICAL-A).** The app's user-visible log level is applied **in-process** at `lifespan.py:157-177` — `ServerConfigManager.load_config()` → map the string level → `SQLiteLogHandler.setLevel(configured_level)`. `main.py`'s `uvicorn.run(...)` (`server/main.py:40`) does **not** pass `log_level` at all. A uvicorn `--log-level` flag only configures uvicorn's **own** loggers; it would NOT change the application's `SQLiteLogHandler` level. Therefore `log_level` must **not** be written into ExecStart — it is materialized into `launch.json` and applied in-process at startup (still restart-required, for simplicity, but via `setLevel`, not the launch line).
2. **`--workers` runs in uvicorn's spawned child processes, not in the app's `sys.argv` (CRITICAL-B).** When `--workers N` is forwarded to `uvicorn.run(workers=N)`, uvicorn's multiprocess supervisor spawns N child worker processes; the application code (`lifespan`) executes in those children, whose argv is NOT a faithful copy of the unit's ExecStart. So "read the live process `sys.argv` and compare to the DB" is **not** a reliable way to know which launch config the running process actually adopted. The source of truth for "what was applied" must instead be the **auto-updater**, which is the only component that actually rewrites ExecStart and restarts.

Additionally, the `--workers`-into-uvicorn wiring has been **inert in every real deployment**: the auto-updater's worker-config writer only matches ExecStart lines containing `"uvicorn"` (`deployment_executor.py:1843`/`:1849`), but the installer's real unit is `code_indexer.server.main`-shaped with **no `uvicorn` substring** (`installer.py:208`) — the only `uvicorn`-shaped ExecStart strings in the repo are test fixtures and the local-dev memo. So this epic's `--workers`→`uvicorn.run` wiring is the **first real multi-worker activation** (see §10 / Story 1 + 3 risk).

## 2. Goal

Persist all launch-affecting settings in the **shared database** (SQLite solo / PostgreSQL cluster), like every other runtime setting, so a single edit propagates to **all** nodes (or the solo server) uniformly, and the auto-updater reconstructs the systemd launch line from the shared value. **One mechanism for solo and cluster.**

**Solo-CLI `--workers` parity (MAJOR-M4).** "One mechanism" requires the solo CLI-managed launch path to honor `workers` too. Today `LifecycleServerManager._start_server_process` (`lifecycle/server_lifecycle_manager.py:375-385`) builds the launch command with ONLY `--host config["host"]` / `--port config["port"]` — it **never passes `--workers`** (verified: the `cmd` list contains only `-m code_indexer.server.main --host … --port …`). So a solo CLI-managed server would silently ignore a multi-worker setting even after this epic. Resolution (folded into Story 3, dependent on its AC0 `main.py --workers` extension): once `code_indexer.server.main` accepts `--workers`, `_start_server_process` MUST append `--workers config["workers"]` (sourced from the same runtime config as host/port, defaulting to 1) so the solo CLI path and the systemd path share one launch contract. Without this, the "one mechanism for solo and cluster" claim is false for the solo CLI-managed path and must be dropped; the epic instead **adds** the flag to keep the claim true.

## 3. Locked decisions

1. **`log_level`** — in scope, **uniform cluster-wide**, but applied **IN-PROCESS** (NOT via ExecStart/uvicorn). It is materialized into `launch.json` and the lifespan log-level read (`lifespan.py:157-177`) sources it from `launch.json` (fallback `config.json` → default), taking effect on the next restart via `SQLiteLogHandler.setLevel`. (CRITICAL-A.)
2. **No per-node overrides** — `workers` (and all launch settings) are strictly uniform across the cluster.
3. **No auto-restart on save** — saving updates shared state + re-materializes each node's `launch.json` (TARGET); applying is the operator's **diagnostics-screen restart** (made cluster-aware).
4. **`host`/`port`/`workers` flow through one `_ensure_launch_config` (ExecStart rewrite); `log_level` is applied in-process** (decision #1). `_ensure_launch_config` rewrites ONLY `--host`/`--port`/`--workers`. `host`/`port` changes require an explicit **server-side** confirmation guardrail (HAProxy backend + firewall "port-lock") validated in `update_config_section` (`routes.py:8756`), not merely a client-side modal (MINOR-C). DSN/`storage_mode`/`cluster.node_id`/`server_dir` stay bootstrap. **Option A prerequisite (CRITICAL-1):** the installer-shape entry point `code_indexer.server.main` (`server/main.py`) today accepts ONLY `--port`/`--host`/`--reload` (`add_argument` at `:15`/`:18`/`:24`; `uvicorn.run(...)` at `:40` — host/port/reload/access_log only, NO `workers`). So if `_ensure_launch_config` appended `--workers` to that ExecStart, the process would fail to start (unrecognized argument). The user-approved fix is **Option A: extend `code_indexer.server.main` to accept `--workers` and forward it to `uvicorn.run(workers=...)`** (NOT `--log-level` — that is in-process, decision #1), preserving the existing `--port`/`--host`/`--reload` behavior. This is folded into **Story 3** as an **ORDERED prerequisite**: the `main.py` CLI `--workers` extension MUST land before `_ensure_launch_config` writes `--workers` into the installer-shape ExecStart (see §7, §14, §12 Story 3).
5. **Design 2** — per-node-local files materialized from / written against the shared DB; the **root auto-updater stays DB-free** (reads the files). The DB is the shared source of truth; the files are a node-private hand-off. **TWO per-node files** (the two-snapshot model, decision #7): `launch.json` (TARGET) + `applied_launch.json` (APPLIED, auto-updater-owned).
6. **No leader orchestration, no new table** — reuse the existing per-node self-restart pattern (same as deploys) via a `launch_restart_generation` field carried on the existing config-version poll, plus per-node files (NOT a new DB table, NOT a shared cidx-meta file).
7. **Two-snapshot TARGET-vs-APPLIED model (resolves Codex CRITICAL×2 + MAJOR, Opus CRITICAL-B/C).** Per-node files under `CIDX_DATA_DIR`:
   - **`launch.json` = TARGET** — server-materialized from the current DB on every save + at startup = `{ host, port, workers, log_level, target_restart_generation }`. (Includes `log_level` so the lifespan in-process read sources it from here.)
   - **`applied_launch.json` = APPLIED** (NEW per-node sidecar, written by the **ROOT auto-updater ONLY**) = `{ host, port, workers, applied_restart_generation }` — exactly what the auto-updater last actually rewrote into ExecStart **and successfully restarted onto**. (`log_level` need not be in the APPLIED sidecar because it is not an ExecStart token — it is applied in-process; carrying it is optional and harmless, but the authoritative APPLIED set for ExecStart reconciliation is host/port/workers + the generation.)

   "Applied" is therefore **owned by the auto-updater** (it actually applied + restarted), NOT inferred from the server process's argv (CRITICAL-B) and NOT inferred from `launch.json == DB` (Codex CRITICAL-3 / old M-C2).

## 4. Verified propagation mechanism (this is why there is no "hole")

**The config-version poll is PostgreSQL-only — it is the CLUSTER propagation path, NOT the solo path.** Both the poll thread and the version check short-circuit when there is no PG pool:

- `ConfigService.start_config_reload(interval_seconds=30)` (`config_service.py:205`) **returns immediately when `self._pool is None`** (`config_service.py:207`) — so in solo/SQLite mode the poll thread is never started.
- `check_config_update()` (`config_service.py:2221`) **returns `False` when `self._pool is None`** (`config_service.py:2226-2227`) — so even if called, the version diff + `_on_change_callbacks` firing never runs in solo mode.

**Cluster (PostgreSQL) — propagation rides the existing poll.** Other nodes are never *pushed* to — they **pull** the shared `server_config.version` on the per-node poll that already runs for every runtime setting:

- `start_config_reload(interval_seconds=30)` starts a per-node daemon thread; started on every node at `lifespan.py:2624` (cluster branch).
- `check_config_update()`: `SELECT version FROM server_config WHERE config_key='runtime'`; if `row["version"] != self._db_config_version` → reload → **fire `self._on_change_callbacks`** (`config_service.py:2236-2244`, Bug #586).
- Every write does `version = version + 1` (`_save_runtime_to_pg:2268-2269`).

So in a cluster: edit on node A → `version++` in the shared PG row → nodes B/C detect it within 30s on their own poll → fire callbacks. We register **cb1 (materialize)** as a config-change callback (it re-materializes `launch.json` (TARGET) on a version edge); and we call **`check_pending_launch_restart()` directly from `_poll_loop` every interval (version-diff-independent)** — it compares `target_restart_generation` (from `launch.json`/raw-row) against `applied_restart_generation` (from `applied_launch.json`). The restart-check is **NOT** a callback (CRITICAL-C1, see §5 below): a callback fires only on the version edge, which is unreachable for the pending-retry after a failed restart or on a node booted behind. No push, no shared file, no new polling infrastructure. This is the exact path every existing runtime setting already uses to go cluster-wide.

**The saving cluster node closes its OWN hole with a DIRECT call (MAJOR-4).** The poll covers *other* nodes (B/C) but NOT the node that performed the save: `_save_runtime_to_pg` (`config_service.py:2259`) updates the **saving node's own** `self._db_config_version` to the just-written version (`config_service.py:2280`), so that node's next `check_config_update()` poll sees **no diff** and its materializer callback **never fires for its own edit** — its `launch.json` would go stale until some unrelated future bump. Fix: call `materialize_launch_config()` **DIRECTLY** after a successful save on the cluster save path too — i.e. `_save_runtime_to_pg` re-materializes the saving node's own `launch.json` synchronously (in addition to the poll-driven re-materialization on nodes B/C). So the direct call is on **BOTH** save paths: `_save_runtime_to_pg` (cluster saving node) AND `_save_runtime_to_sqlite` (solo). The poll remains the cross-node propagation path for the non-saving cluster nodes; the direct call covers the saving node in every mode.

**Solo (SQLite) — propagation is a DIRECT call on the save path (NOT the poll).** Because the poll never runs without a PG pool, `_on_change_callbacks` never fire on a solo save, so the materialization cannot ride the poll. Instead, the solo save path calls `materialize_launch_config()` **directly** so `launch.json` is re-materialized on **every** solo save — not just at startup. The direct call lives on the SQLite runtime-save path (`_save_runtime_to_sqlite`, `config_service.py:2376`). (Solo is single-node, so there is no cross-node hole to close — only the local file must be refreshed.) This is the **same direct-call mechanism** the saving cluster node uses on `_save_runtime_to_pg` (above) — one direct-call hook applied to both save paths.

**The pending-retry is an EXPLICIT per-poll method, NOT a `_on_change_callbacks` hook (CRITICAL-C1).** The diagram's "cb2 EVERY poll while target_gen > applied_gen" must NOT be implemented as a Bug #586 `_on_change_callbacks` callback. The callback registry fires ONLY inside `if row["version"] != self._db_config_version` (`config_service.py:2236-2239`) — a one-shot edge that fires once per version change. After a failed restart, or on a node booted behind (where `_load_runtime_from_pg` already set `_db_config_version` to the current version at `config_service.py:2345`), there is **no further version change**, so a callback-based retry is **unreachable** — "every poll while target>applied" can never fire again. **Resolution:** add an explicit method, e.g. `check_pending_launch_restart()`, **called directly from the poll loop** (`start_config_reload`'s `_poll_loop`, `config_service.py:213-219`, which already runs `check_config_update()` every interval) on **every** interval, INDEPENDENT of the version diff. It (re-materializes `launch.json` then) compares the TARGET generation (raw-row snapshot / `launch.json`) against APPLIED (`applied_launch.json`) and, while `target > applied`, writes `restart.signal` (M-S1: only on a successful materialize). This is **cluster-only** — the poll runs only with a PG pool (`start_config_reload` returns early when `_pool is None`, `config_service.py:207`); solo uses the direct single-node restart (no generation), per §8. The materialize-on-version-bump for non-saving nodes (cb1) REMAINS a Bug #586 callback (it is genuinely a version-edge action); only the pending-retry restart-check (cb2) moves to the explicit per-poll method. (See §10 "Pending-retry…" and §6.)

**`_ensure_launch_config` has a MODE (resolves the apply-vs-deploy ambiguity):**

- **APPLY mode** — invoked from the **restart-signal path** (`service.py:139`). Source = **`launch.json` (TARGET)**. Strict sequence with applied-sidecar write sequencing (MAJOR-M5): (1) `_ensure_launch_config(APPLY)` rewrites ExecStart `--host/--port/--workers` from `launch.json` + `daemon-reload`, and **returns the TARGET snapshot it materialized into ExecStart ONLY on a successful rewrite + reload** (a falsy/None return signals the ensure failed); (2) `service.py` calls `restart_server()`; (3) `applied_launch.json` is written = the target values + `applied_restart_generation = target_restart_generation` **ONLY IF BOTH the ensure succeeded AND `restart_server()` returned success**. If the ensure fails → no `restart_server()` is wastefully run on a stale unit and no `applied_launch.json` write; if the restart fails → no `applied_launch.json` write (so the per-poll pending-retry, C1, re-signals next interval). The truth source for "applied" is "the auto-updater actually rewrote ExecStart AND the restart succeeded" — NOT the server's argv (CRITICAL-B), NOT `launch.json == DB`. NOTE: the current handler ignores `_ensure_workers_config`'s return value (`service.py:138-139`, restart unconditionally at `:148`); the new flow MUST check the ensure return before restarting and gate the `applied_launch.json` write on both results.
- **DEPLOY mode** — invoked from `execute()` Step 3 (`deployment_executor.py:2890`), the routine code-deploy path (`git pull` → `pip install` → ensure-config → restart). Source = **`applied_launch.json` (APPLIED)**, NOT the current DB / `launch.json` / TARGET. So a routine code deploy preserves the **last operator-applied** launch config and does NOT over-apply a saved-but-unconfirmed change (Opus CRITICAL-C, decision #3 "no auto-restart on save"). **DEPLOY MUST NEVER read `launch.json` (TARGET) — MAJOR-M1.** The fallback chain is `applied_launch.json` → `config.json` → `ServerConfig` defaults, deliberately omitting `launch.json`: on first boot / no-APPLIED-yet, or when `applied_launch.json` is corrupt/malformed, falling through to TARGET would re-open the over-apply (decision #3) in exactly that window. **Corrupt/malformed `applied_launch.json` in DEPLOY mode → log a WARNING and preserve the CURRENT ExecStart** (parse the live unit's existing `--host/--port/--workers` and keep them); do NOT adopt TARGET and do NOT silently fall through to `config.json`/defaults if a working ExecStart already exists. (APPLY mode's fallback chain, by contrast, IS allowed to read `launch.json` → `config.json` → defaults, because APPLY's whole job is to apply the TARGET.)

**Cluster diagnostics-restart bump must not advance the bumping node's local `_db_config_version` (MAJOR-M3).** The atomic in-SQL generation increment (§6 req. 2) bumps the shared `version` in the same statement. Unlike `_save_runtime_to_pg` — which, after its write, advances the **saving** node's own `_db_config_version` to the just-written value (`config_service.py:2280`) so its own poll never sees the diff — the diagnostics bump path MUST **leave the bumping node's local `_db_config_version` UNCHANGED** so the bumping node's own next poll detects the new version and (via cb1) re-materializes its `launch.json` with the new `target_restart_generation`, and (via the explicit per-poll `check_pending_launch_restart()`, C1) self-signals its own restart. (The explicit per-poll check is generation-driven and version-diff-independent, so it self-signals regardless; but leaving `_db_config_version` unchanged also makes cb1 re-materialize the bumping node's TARGET, which the check needs.) **Alternatively**, the bump endpoint may directly invoke `check_pending_launch_restart()` on the local node immediately after the bump. The spec mandates the FIRST option (do not advance `_db_config_version` on the bump) as the canonical implementation, with the direct-invoke as a permitted belt-and-suspenders add-on. A test MUST assert the operator's own (bumping) node restarts (its `restart.signal` is written after the bump, without waiting for an external version change it would otherwise mask).

**Flow — edit:** save → DB `version++` → **cluster:** the **saving** node re-materializes its own `launch.json` (TARGET) via a DIRECT call on `_save_runtime_to_pg` (its own poll sees no diff — MAJOR-4), and every **other** node's 30s poll re-materializes its own `launch.json`; **solo:** the SQLite save path re-materializes `launch.json` directly (no poll — §4) (NO restart in either mode; decision #3).

**Flow — apply:** operator clicks diagnostics restart → **cluster:** the endpoint performs the ATOMIC in-SQL increment of `launch_restart_generation` (`version++`, §6) **ONLY**, **without advancing the bumping node's local `_db_config_version`** (MAJOR-M3) so the bumping node detects its own bump. **In cluster systemd mode the endpoint does NOT also invoke the legacy `_delayed_restart` node-local `restart.signal` write (M2):** the existing diagnostics restart (`web/routes.py:11383` `_delayed_restart`) writes a node-local `restart.signal` unconditionally in systemd mode (`:11419`) — but that direct write must be SUPPRESSED in cluster mode, otherwise the bumping node would restart via two asymmetric paths (a direct local signal AND the uniform per-poll mechanism), undercutting the uniform per-poll mechanism (M3). Instead the bumping node restarts SOLELY via its own per-poll `check_pending_launch_restart()`, exactly like every other node. Then every node's poll runs the EXPLICIT per-poll method `check_pending_launch_restart()` (C1 — called from `_poll_loop` every interval, version-diff-INDEPENDENT, NOT a Bug #586 callback), and **while `target_restart_generation > applied_restart_generation`** (a pending-retry, evaluated on every poll regardless of whether the version changed — Codex MAJOR) it **first** re-materializes `launch.json` (TARGET) and **only on a successful materialize** writes its local `restart.signal` (M-S1). The per-poll check does **NOT** record applied — the auto-updater records `applied_launch.json` post-restart-success (Codex CRITICAL "applied marked before restart succeeds"). The auto-updater runs `_ensure_launch_config` in **APPLY mode** (rebuild ExecStart from `launch.json`) → `systemctl restart` → writes `applied_launch.json` **only if both the ensure and the restart succeeded** (MAJOR-M5). Because `check_pending_launch_restart()` re-checks `target > applied` every cycle independent of any version change, a failed materialize/restart simply retries on the next poll **without another DB bump** (Codex MAJOR) — this is precisely why it cannot ride the version-edge callback (C1). Staggered per-node (poll offsets), same uncoordinated pattern deploys already use. **Solo:** materialize then the existing single-node restart path — solo/dev **RETAINS** the legacy `_delayed_restart` (`web/routes.py:11383`) systemd-mode local `restart.signal` write (`:11419`) / dev `os.execv` re-exec (M2 cluster suppression does NOT apply to solo); the auto-updater still owns `applied_launch.json` in solo systemd; no generation bump (MAJOR-6).

## 6. Data model — NO new table, NO new migration

- **Source of truth:** the existing `server_config` runtime row. `workers`/`log_level`/`host`/`port` become ordinary runtime JSON keys (removed from `BOOTSTRAP_KEYS`). Plus a coordination field `launch_restart_generation` (int, default 0) in the same runtime JSON — not a user-editable setting, bumped only by the diagnostics-restart endpoint in cluster mode.
- **No schema migration** (server_config exists; all additions are JSON fields). The one-time `config.json`→DB value seed is handled by the existing idempotent `initialize_runtime_db()` path (`config_service.py:165`, wired unconditionally at `service_init.py:231`).
- **Per-node files — pinned to the AUTO-UPDATER's shared data dir (MAJOR-M2), atomic tempfile+`os.replace`:** both files live in `_cidx_data_dir` (`deployment_executor.py:31-33` = `Path(os.environ.get("CIDX_DATA_DIR", str(Path.home()/".cidx-server")))`), i.e. **`RESTART_SIGNAL_PATH.parent`** (`deployment_executor.py:55`) — the SAME directory and SAME resolution Bug #879 used to align the restart-signal path across the two OS users (server runs as `code-indexer`, auto-updater as `root`). They MUST **not** be written under the server's `CIDX_SERVER_DATA_DIR` (`server_data_dir`), because the ROOT auto-updater (which reads `launch.json` in APPLY and reads/writes `applied_launch.json`) resolves IPC paths via `CIDX_DATA_DIR`, not the server's data dir. ### `launch_restart_generation` representation + atomicity (NOT a dataclass round-trip)

`launch_restart_generation` is a **raw JSON key inside `server_config.config_json`** — it is **not** a `ServerConfig` dataclass field, and it must be handled entirely **outside** the `ServerConfig`/`asdict()` round-trip.

**Why exclusion ALONE is NOT enough (CRITICAL-2 — corrects the prior "exclusion is sufficient" claim).** The normal settings-save serializes the whole `ServerConfig` via `_extract_runtime_dict(config)` (`asdict(config)` filtered by `BOOTSTRAP_KEYS`, `config_service.py:2248-2257`) and overwrites `config_json` **wholesale** — `UPDATE server_config SET config_json = %s, version = version + 1 ...` (PG, `config_service.py:2266-2272`, the actual `%s` write at `:2268`) / the equivalent `config_json = excluded.config_json` (SQLite, `_save_runtime_to_sqlite`, `config_service.py:2383`). Because the write is a **full-blob replace**, simply *excluding* `launch_restart_generation` from `_extract_runtime_dict` does **not** preserve it — it means the next settings-save writes a `config_json` blob that **does not contain the key at all**, i.e. the save **DELETES** the generation. Worse, the key cannot survive a `ServerConfig` round-trip either: `_dict_to_server_config()` strips any key not in `{f.name for f in dc_fields(ServerConfig)}` (`config_manager.py:2117-2121`), so a non-dataclass top-level key is dropped the moment it passes through `ServerConfig`. Therefore the generation must be **read and written via raw-row SQL only**, never via `ServerConfig`.

Three hard requirements (all generation reads via a **single raw-row snapshot helper** — Opus MAJOR-B):

1. **The normal settings-save must explicitly PRESERVE the existing raw `launch_restart_generation` in the SAME UPDATE, atomically against the current row** (never a read-then-overwrite that can lose a concurrent bump — Codex CRITICAL). It must NOT blindly overwrite `config_json` with the `asdict()`-derived blob (which lacks the key). **Use a TARGETED re-inject of ONLY `launch_restart_generation` — NOT a blanket merge (Codex nit).** A blanket `config_json = config_json || %s::jsonb` would carry through **every** pre-existing key in the current row, including arbitrary unknown/legacy keys that the full-blob `asdict()`-filtered save **currently drops** by design (`_dict_to_server_config` strips any top-level key not in the `ServerConfig` dataclass fields, `config_manager.py:2117-2121`). Preserving those would silently change the save's cleanup semantics. So the preserve must re-inject the **single** `launch_restart_generation` key, not splice the whole prior blob. Two acceptable implementations:
   - **targeted in-SQL re-inject against the current row (PG):** write the new `asdict()`-derived runtime blob via `%s::jsonb`, then `jsonb_set` ONLY the `launch_restart_generation` key from the **current** row back onto it in the SAME `UPDATE` — e.g. `config_json = jsonb_set(%s::jsonb, '{launch_restart_generation}', to_jsonb(COALESCE((config_json->>'launch_restart_generation')::int, 0)))` — so exactly one key is carried through (read from the current row inside the UPDATE, so a concurrent diagnostics bump cannot be lost), and all other dropped-by-design keys stay dropped; or
   - **`SELECT … FOR UPDATE` then write (PG) / `BEGIN IMMEDIATE` single transaction (SQLite):** read the current raw `config_json->>'launch_restart_generation'` under a row lock, splice **only that one key** into the `asdict()`-derived runtime dict, then `UPDATE … config_json = %s` — all in one transaction so no interleaving bump is lost.

   Either way the post-save row MUST still carry whatever `launch_restart_generation` it held before the save, **even if a diagnostics bump committed concurrently**, and MUST NOT resurrect any other key the wholesale save would otherwise drop. (Excluding the key from `_extract_runtime_dict` is still correct — it keeps the generation out of the user-editable serialized surface — but it is NOT sufficient on its own; the atomic targeted re-inject step is what actually keeps the one key alive across the full-blob overwrite.)
2. **The diagnostics-restart bump is a single ATOMIC in-SQL increment** that both increments the generation and bumps `version`, computed against the CURRENT row (never read-modify-write through `ServerConfig`):
   - **PostgreSQL (cluster):**
     ```sql
     UPDATE server_config
        SET config_json = jsonb_set(
              config_json,
              '{launch_restart_generation}',
              (COALESCE((config_json->>'launch_restart_generation')::int, 0) + 1)::text::jsonb),
            version = version + 1
      WHERE config_key = 'runtime';
     ```
   - **SQLite (solo):** the equivalent atomic read-increment-write performed **inside one `BEGIN IMMEDIATE` transaction** (read current generation, increment, write back the JSON, bump `version` — all in a single transaction so no interleaving save can lose the bump). **The SQLite atomic-bump shape exists for UNIT-TEST modeling of raw-row atomicity ONLY (FIX-5 / Codex nit).** In real solo operation the generation is **NEVER** bumped — solo uses the direct single-node restart path (MAJOR-6; the diagnostics restart in solo materializes then signals via the existing single-node `restart.signal` / dev path, no generation). The symmetric SQLite helper is provided so the raw-row atomicity contract (req. 1/2/3) can be exercised against real SQLite in tests; it is not invoked by the solo diagnostics-restart route. This resolves the apparent AC1-vs-AC7 tension in Story 4: AC1's "SQLite-equivalent bump" is the test-modeling helper, AC7 is the real solo behavior (no bump).
3. **All reads of `launch_restart_generation` (the restart-check callback AND the startup read) use a SINGLE RAW-ROW snapshot helper** that reads the four launch values **and** the generation **together** from ONE row read (Opus MAJOR-B) — e.g. `SELECT config_json FROM server_config WHERE config_key='runtime'` then extract `host`/`port`/`workers`/`log_level`/`launch_restart_generation` from that one snapshot — and NEVER read the generation off a `ServerConfig` instance (which can never carry it, per `config_manager.py:2117-2121`). **NULL/absent generation → `COALESCE 0`** so the pre-first-bump case is covered (Opus MAJOR-C). The generation is owned end-to-end by raw SQL: atomic-increment (req. 2) for the bump, single raw-row snapshot (req. 3) for reads, atomic preserve/merge (req. 1) so the user-facing settings-save never erases it.

## 8. Cluster vs solo — one materializer, two trigger paths, auto-updater-owned APPLIED

The shared DB row is the source of truth; `launch.json` (TARGET) is the per-node derived hand-off in BOTH modes; `applied_launch.json` (APPLIED) is auto-updater-owned in BOTH modes. What differs is **how re-materialization is triggered** and **whether the generation is bumped**:

| Aspect | Cluster (PostgreSQL) | Solo (SQLite) |
|--------|----------------------|---------------|
| Re-materialize TARGET trigger | **saving** node: a **direct call** to `materialize_launch_config()` on `_save_runtime_to_pg` (its own poll sees no diff — MAJOR-4); **other** nodes: the 30s config-version **poll** fires `_on_change_callbacks` → `materialize_launch_config()` | a **direct call** to `materialize_launch_config()` on the save path (`_save_runtime_to_sqlite`); the poll never runs (`start_config_reload`/`check_config_update` short-circuit on `_pool is None`) |
| Cross-node propagation | YES — non-saving nodes pull the `version` bump; the saving node covers itself via the direct call | N/A — single node |
| Generation bump on diagnostics restart | atomic SQL increment of `launch_restart_generation` (§6), WITHOUT advancing the bumping node's local `_db_config_version` (MAJOR-M3) so the bumping node self-detects; each node's explicit per-poll `check_pending_launch_restart()` (C1 — NOT a Bug #586 callback) runs every interval and, while `target_gen > applied_gen`, materializes-then-signals (pending-retry) | NO generation bump (MAJOR-6) — the single node materializes the current launch config, then uses the existing single-node `restart.signal` / dev restart path |
| Auto-updater ExecStart rebuild | APPLY mode reads `launch.json`; on restart success writes `applied_launch.json` (per-node) | identical — one node; auto-updater still owns `applied_launch.json` in solo systemd (CLI-dev re-exec path also re-materializes) |
| "Applied" truth source | `applied_launch.json` (auto-updater-owned, post-restart-success) — NOT argv (CRITICAL-B), NOT `launch.json == DB` | same — `applied_launch.json` |

Same materializer, same `launch.json`/`applied_launch.json` files, same auto-updater rebuild, same `restart.signal` apply path. Solo is the degenerate one-node case, but its re-materialization rides the **save path's direct call**, not the (absent) poll, and it never bumps the generation.

## 9. Migration + rolling-upgrade safety

- One-time `config.json`→DB value seed via existing `initialize_runtime_db()` (idempotent).
- **Stays bootstrap:** `postgres_dsn`, `storage_mode`, `cluster`, `server_dir`. (`host`/`port` move to runtime but the systemd unit keeps a working value until `_ensure_launch_config` runs.)
- **Critical back-compat:** do NOT strip `workers`/`log_level`/`host`/`port` from `config.json` for one release cycle — old nodes keep reading the file; new nodes prefer DB/`launch.json`; the file is the old-node fallback. A later release (Story 6) removes the file copies once all nodes are new. **This requires TWO mechanisms this release (MAJOR-5), not just the strip-guard:** (1) the **strip-guard** on `_strip_config_file_to_bootstrap()` (C-A5 below); AND (2) the **config.json WRITE-path inclusion** — the normal `save_config()` writes only `_extract_bootstrap_dict(config)` to `config.json` on BOTH the PG path (`config_service.py:2475-2476`) and the SQLite path (`config_service.py:2479-2480`), so once the four keys leave `BOOTSTRAP_KEYS` a normal save writes a `config.json` WITHOUT them — silently dropping the old-node fallback on the first settings-save even if the first-boot strip was guarded. The write path must therefore INCLUDE `TRANSITION_PRESERVE_KEYS = {workers, log_level, host, port}` on BOTH save paths this release. Story 6 removes BOTH the strip-guard AND the write-path inclusion next release.

### C-A5 — `_strip_config_file_to_bootstrap()` would strip the four keys on first boot (existing code; MUST be guarded)

`initialize_runtime_db()` already calls `_strip_config_file_to_bootstrap()` (`config_service.py:2348`, invoked at `:176` and `:188`) on first boot. That method **deletes every key NOT in `BOOTSTRAP_KEYS` from `config.json`** (writes `_extract_bootstrap_dict(config)` at `:2369-2370`). The moment Story 1 removes `workers`/`log_level`/`host`/`port` from `BOOTSTRAP_KEYS`, this **existing** strip code will delete those four keys from `config.json` on the very first boot — destroying the "keep `config.json` fallback for one release" plan **before** any old node can read it.

**Required fix (Story 1) — TWO parts (MAJOR-5):** (a) add a **documented transition allow-list** so `_strip_config_file_to_bootstrap()` preserves these four transition keys for THIS release — i.e. the strip set is `(all keys) − BOOTSTRAP_KEYS − TRANSITION_PRESERVE_KEYS`, where `TRANSITION_PRESERVE_KEYS = {workers, log_level, host, port}`; AND (b) make the normal `save_config()` config.json WRITE path INCLUDE `TRANSITION_PRESERVE_KEYS` on BOTH save paths (`config_service.py:2475-2476` PG, `:2479-2480` SQLite) so the four keys stay written/updated in `config.json` on every settings-save (the strip-guard alone only protects the first-boot strip, not the per-save write). The four keys stay in `config.json` as the old-node fallback even though they are no longer bootstrap keys. **Story 6 (next release)** removes BOTH the transition allow-list AND the write-path inclusion, and lets the strip delete them — the file copies disappear once all nodes are new.

- No schema migration; no DROP/RENAME. `launch.json`/`applied_launch.json` are inert to old nodes (they never read them).

## 10. Failure modes

- **DB down at restart:** auto-updater reads `applied_launch.json` (DEPLOY) / `launch.json` (APPLY) → `config.json` → defaults. **DEPLOY never reads `launch.json` / TARGET (MAJOR-M1).** Root never touches the DB. No new failure mode. **NOTE (transition vs post-cleanup — FIX-1):** the `config.json` rung shown here is the **transition-release** behavior (Stories 1-5, while the `config.json` launch copies still exist). After Story 6 removes those copies, the `config.json` rung is dropped: DEPLOY becomes `applied_launch.json` → parse/preserve the current live ExecStart → `ServerConfig` defaults (still NEVER `launch.json`/TARGET), and APPLY becomes `launch.json` → `ServerConfig` defaults — see §12 Story 6.
- **`launch.json` write fails:** fail-soft WARNING (mirrors v10.160.0 + Bug #1182), retried next poll (the explicit per-poll `check_pending_launch_restart()` re-fires while `target_gen > applied_gen`); an APPLY restart falls back to `config.json` → defaults (NOT TARGET on the DEPLOY path).
- **Corrupt/malformed `applied_launch.json` in DEPLOY (MAJOR-M1):** log a WARNING and **preserve the CURRENT live-unit ExecStart** (parse and keep its existing `--host/--port/--workers`). Do NOT adopt `launch.json`/TARGET (that would over-apply a saved-but-unconfirmed change), and do NOT blindly fall through to `config.json`/defaults if a working ExecStart already exists. APPLY mode, by contrast, IS allowed to read `launch.json` → `config.json` → defaults (applying the TARGET is APPLY's job).
- **M-S1 — materialize-before-signal ordering (fail-soft on the per-poll check):** the explicit per-poll `check_pending_launch_restart()` MUST run `materialize_launch_config()` **first** and MUST NOT write `restart.signal` if materialization failed. Order: (1) materialize `launch.json` (TARGET) for the target generation; (2) **only if** `launch.json` was successfully written, write `restart.signal`. The check does **NOT** record applied (the auto-updater records `applied_launch.json` post-restart-success — Codex CRITICAL). If materialize failed, skip the restart-signal and log a WARNING — never signal a restart that would rebuild ExecStart from a stale/absent `launch.json`. The next poll retries materialization (and, finding `target_gen > applied_gen` still true, re-signals). A source-order/behavioral test must enforce this ordering.
- **Pending-retry on EVERY poll via an EXPLICIT method, not a version-edge callback (CRITICAL-C1 / Codex MAJOR).** `check_pending_launch_restart()` is called from the poll loop **every interval, independent of any version change** (it is NOT a Bug #586 `_on_change_callbacks` hook — those fire only inside `if version != _db_config_version`, `config_service.py:2236-2239`, a one-shot edge that is unreachable after a failed restart or on a node booted behind whose `_db_config_version` was already set at `config_service.py:2345`). It compares the local TARGET generation (from `launch.json`/raw-row snapshot) against `applied_restart_generation` (from `applied_launch.json`) on **every** poll. As long as `target > applied`, it (re-materializes then) re-writes `restart.signal`. So a failed materialize, a failed restart, or an auto-updater that has not yet run will **retry on the next poll without requiring another DB bump**. The poll interval (`30s`) stays below `RESTART_SIGNAL_STALENESS_THRESHOLD` (`120s`, `deployment_executor.py:58`) so the refreshed signal never ages into a stale-discard. Once the auto-updater restarts successfully and writes `applied_launch.json` with `applied == target`, the comparison goes false and the loop stops (no spurious restart loop). A regression test MUST cover: version unchanged + `target > applied` → a second poll re-signals (proves the retry does not depend on a version diff). **Stuck-node visibility (FIX-6 / Opus minor):** when `target > applied` has persisted across **many consecutive polls** (e.g. `> 10` cycles ≈ 5 min at the 30s poll interval), `check_pending_launch_restart()` emits **ONE rate-limited WARNING** so a stuck or stopped auto-updater (a node that keeps re-signalling but never converges) surfaces via the existing log-audit gate (Story #1122). This is a single minimal log line, NOT a status view — the per-node applied-generation status view was never requested and remains out of scope (§13); the WARNING is rate-limited (one per stuck episode, not per poll) so it does not spam the log store.
- **M-C2 superseded — "applied" is auto-updater-owned, NOT boot-inferred (CRITICAL-3 / CRITICAL-B resolved).** The prior design tried to verify on startup whether the LIVE process was running the latest generation by reading its argv. That is **wrong twice over:** (a) `launch.json == DB` does NOT prove the running process applied those values, because `_ensure_launch_config` only runs in the auto-updater restart-signal path (`service.py:139`), NOT at every boot — a node down during generation 5 boots on the OLD unit; and (b) `--workers` runs in uvicorn's **spawned children**, so the server process's `sys.argv` is NOT a faithful record of the applied launch config (CRITICAL-B). **Resolution:** make APPLIED auto-updater-owned. On startup the node does NOT mark applied from `launch.json == DB` and does NOT read its own argv. It simply materializes `launch.json` (TARGET) from the current DB and reads `applied_launch.json` (whatever the auto-updater last wrote). A node booted on an old unit therefore has `applied_restart_generation < target_restart_generation`, so its restart-check poll **re-signals** (pending-retry) → the auto-updater rebuilds ExecStart in APPLY mode, restarts, and writes a fresh `applied_launch.json` with `applied == target`. No argv reading, no `launch.json == DB` inference, no race window where a derived-file match falsely marks applied. (If `applied_launch.json` is absent on a brand-new node whose unit already carries correct ExecStart, the first restart-check still re-signals harmlessly; the auto-updater's idempotent value-aware rewrite is a no-op on ExecStart and simply records `applied_launch.json` after the restart — converging in one cycle.)
- **Two-admin race:** `version` monotonic last-writer-wins (existing `server_config` semantics); `launch_restart_generation` is bumped by atomic SQL increment (§6 req. 2) and PRESERVED by an atomic in-SQL merge / `FOR UPDATE` on the settings-save (§6 req. 1) so concurrent bumps cannot lose each other (Codex CRITICAL); a node behind converges on the next poll.
- **Staggered apply:** nodes restart within their poll offsets (≤30s config poll + ≤60s auto-updater poll); HAProxy health-checks route around a restarting node; no full outage (same as deploys today).

The confirm flag must be REQUIRED and validated in `update_config_section` (`routes.py:8756`) server-side, not merely a client-side modal. DSN/storage_mode/node_id never leave bootstrap.
