FIXED v11.23.0 (#1313). The temporal metadata store (`hash_prefix -> point_id` map, `temporal_metadata_store.py`, write-only at runtime) was a per-shard SQLite-WAL DB on the shared golden-repos NFS mount, committed+WAL-checkpointed per-commit by 8 threads â†’ ~25x slowdown (py-spy proven) and a Cluster-Aware-State violation. Now backend-pluggable: `TemporalMetadataBackend` protocol + `TemporalMetadataSqliteBackend` (CLI/solo, unchanged) + `TemporalMetadataPostgresBackend` (cluster; migration 033, table `temporal_metadata`, `collection_key = sha256(str(collection_path))[:32]`, batch `INSERT ... ON CONFLICT`, `SET LOCAL synchronous_commit=off`). Selection via a process-local factory registry (coalescer_registry idiom).

THE LOAD-BEARING GOTCHA (took 5 Codex rounds to fully close): cluster temporal indexing does NOT run in the server process â€” it runs in child `cidx index --index-commits` subprocesses (launched by golden_repo_manager/refresh_scheduler/repos/activated_repo_index_manager). A process-local in-memory registry set in the server lifespan is EMPTY in those children, so `get_temporal_metadata_backend_factory()` returns None and the facade silently falls back to SQLite-on-NFS â€” making a lifespan-only fix INERT on the real hot path (same failure class as the "wiring is POST-HOC, removing it makes the fix INERT" invariants in CLAUDE.md). Fix: a path-only IPC contract `CIDX_TEMPORAL_PG_BOOTSTRAP_DIR` (carries the server dir PATH, never the DSN â†’ not on argv/env; the child re-reads `postgres_dsn` from bootstrap config.json and installs the PG factory before any `TemporalMetadataStore` construction; fail-loud `sys.exit(1)`, never SQLite fallback in cluster). ALL FIVE server-side temporal launch sites must pass it in postgres mode (post-clone registration, scheduled refresh, add_golden_repo_index temporal, provider temporal job [merge into provider API-key env via base_env], activated-repo temporal). A source-scan enumeration guard test (`test_temporal_pg_env_wiring_site_enumeration_guard_1313.py`) fails CI if a new `--index-commits` launch site is added without `build_temporal_child_env`. Server lifespan installs the real factory or a fail-loud POISON factory (raise-on-use) in postgres mode â€” never leaves the registry unset (unset == the SQLite sentinel).

Practical notes for future validation:
- Validate cluster temporal metadata against PostgreSQL (`SELECT count(*) FROM temporal_metadata`), NOT SQLite. Existing pre-fix SQLite-on-NFS temporal indexes need NO forced re-index (store is write-only at query time); PG rows populate on next refresh; orphaned `temporal_metadata.db` files remain on NFS (ignored, optionally sweep later).
- The DECISIVE "is the fix engaging PG?" tell on staging: NO `temporal_metadata.db`/`-wal`/`-shm` file reappears on the NFS golden-repos mount during a temporal index (a pre-fix repo still carries its old NFS .db â€” use as a control). Positive: PG rows grow + commits/s >> ~0.1-0.3 baseline (staging measured ~26 commits/s for 200 commits).
- Measuring temporal throughput: count `distinct commit_hash` in PG, NOT the progress bar's per-quarterly-shard "N/N commits" counter (a display landmine that undercounts the true total).
